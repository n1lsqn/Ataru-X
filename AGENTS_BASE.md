# AGENTS_BASE.md - 共通開発ガイドライン & ルール

本ドキュメントは、プロジェクトを跨いで共通して使用できるエージェント開発ガイドラインおよびベストプラクティスを定義します。

---

## 1. 基本方針

- **プレースホルダーの禁止**: 未実装コードや `TODO` コメントを残したままタスクを完了としない。
- **検証の実施**: コード変更後は、ビルド・リンター・テストを実行して自己修復し、検証完了後に完了報告する。
- **変更スコープの遵守**: 依頼範囲外のファイルは変更しない。変更が必要な場合は事前にユーザーに確認する。
- **使用言語**: ユーザーへの報告・説明はすべて日本語で行う。

---

## 2. タスク管理フロー

### タスク開始時
1. **目的・スコープ・完了条件の合意**: 着手前に必ずユーザーと完了条件を整合する。
2. **既存設計の把握**: 関連ファイルを先に読み込み、既存のコード設計・命名規則・意図を理解する。
3. **質問の集約**: 不明点は小出しにせず、開始前にまとめてクリアにする。

### タスク完了時
1. **日本語でのサマリー報告**: 何を・なぜ・どう変更したかを明確に報告する。
2. **作業ログの更新**: `agents.md` 内の「作業ログ」セクションに成果内容を追記する。
3. **コミットの実行**: 以下のフォーマットに従ってコミットを行う。

```bash
git add .
git commit -m "<type>(<scope>): <変更要約>"
```
* **Type一覧**: `feat` / `fix` / `refactor` / `style` / `test` / `docs` / `chore` / `perf`

---

## 3. コーディング規約 & セキュリティ

- **型安全性の担保**: `any` の使用は原則禁止とし、`unknown` と型ガードを組み合わせる。
- **機密情報の管理**: シークレットや API キーはコードにハードコードせず、必ず `.env` 等の環境変数で管理する。
- **バリデーション**: ユーザー入力値はクライアント側だけでなく、必ずサーバーサイドでも検証する。
- **単一責任の原則**: 1つのファイル、1つのコンポーネントの責務は1つに留める。
- **コミット制限**: `.env` などの環境設定ファイルはリポジトリにコミットしない（`.gitignore` に追加する）。

## 4. Docker Compose ビルド高速化手法（開発ベストプラクティス）

Docker Compose を用いた開発プロセスを高速化するため、以下のプラクティスを必ず適用する。

### 1) `.dockerignore` による不要ファイルの除外
コンテナイメージビルド時に不要なファイル（ローカルの `node_modules`、`.next`、`dist`、ビルドログ等）がコンテナにコピーされ、キャッシュが無効化されるのを防ぐ。

- **バックエンド用例 (`.dockerignore`)**:
  ```text
  node_modules
  dist
  .env
  *.log
  .git
  ```
- **フロントエンド用例 (`.dockerignore`)**:
  ```text
  node_modules
  .next
  out
  .env*.local
  *.log
  .git
  ```

### 2) BuildKit のキャッシュマウント（`--mount=type=cache`）
Dockerfile で `npm install` または `yarn install` を行う際、BuildKit のマウント機能を利用してホスト側のパッケージマネージャキャッシュをコンテナに共有する。パッケージの変更があった場合でも差分のみを高速で取得できる。

- **Dockerfile での実装例**:
  ```dockerfile
  # package.json をコピーした上で、キャッシュマウントを指定してインストール
  COPY package*.json ./
  RUN --mount=type=cache,target=/root/.npm npm install --legacy-peer-deps
  ```

## 5. 作業ログ記載フォーマット

`agents.md` などの作業ログには以下の形式で時系列に追記する。

```markdown
## YYYY-MM-DD <作業タイトル>
- [変更種別] 実装内容の要約
```
